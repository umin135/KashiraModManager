# 모드 오버라이드 전략 분석

KatanaEngine 게임에서 에셋을 교체하는 네 가지 전략과 각각의 장단점을 정리한다.

---

## 전략 비교표

| 전략 | 신뢰도 | 비파괴 | 크기 제한 | 새 에셋 추가 | 복잡도 |
|------|--------|--------|-----------|-------------|--------|
| A. IDRK 인플레이스 패치 | ✅ 검증됨 | ❌ | 동일 크기만 | ❌ | 낮음 |
| B. RDB 엔트리 리다이렉트 | ✅ 높음 | △ (원본 백업) | 제한 없음 | ❌ | 중간 |
| C. RDB 중복 추가 (last-wins) | ⚠️ 가설 | ✅ | 제한 없음 | ❌ | 낮음 |
| D. 완전 새 에셋 등록 | ⚠️ 미검증 | △ | 제한 없음 | ✅ | 매우 높음 |

---

## 전략 A: IDRK 인플레이스 패치

기존 .fdata 파일 내의 특정 IDRK 블록을 **직접 덮어쓴다**.
현재 모딩 커뮤니티에서 주로 사용하는 방식.

### 절차

```
1. RDB에서 target file_ktid → (fdata_id, fdata_offset, size_in_cont) 획득
2. RDX에서 fdata_id → 파일명 획득 → .fdata 파일 열기
3. 새 에셋을 zlibext로 압축
4. 새 IDRK 블록 구성:
   - total_block_size   = 0x58 + compressed_size  (원본과 동일해야 이상적)
   - compressed_size    = 새 payload 크기
   - uncompressed_size  = 새 에셋 원본 크기
5. .fdata 파일의 fdata_offset 위치에 덮어쓰기
```

### 크기 제약 문제

새 에셋이 기존보다 클 경우:
- 인접한 IDRK 블록을 덮어써서 다른 에셋이 손상됨
- 해결책: 새 에셋을 충분히 작게 압축하거나, 후술하는 전략 B/C 사용

새 에셋이 기존보다 작을 경우:
- IDRK 헤더의 `total_block_size`를 원본 크기로 유지하면 나머지는 패딩으로 처리
- 단순하게 동작 가능 (엔진이 `total_block_size`를 믿는다면)

### 백업 필수

원본 .fdata 파일을 반드시 백업한다. 실수 시 복원이 불가능하다.

---

## 전략 B: RDB 엔트리 리다이렉트

기존 RDB 엔트리의 위치 메타를 수정하여 새 .fdata 파일의 새 블록을 가리키게 한다.

### 절차

```
1. 새 모드 .fdata 파일 생성 (새 IDRK 블록 포함)
2. 새 파일을 RDX에 등록: 새 fdata_id(미사용) + 파일명 해시
3. RDB의 기존 엔트리 위치 메타 수정:
   - fdata_offset → 새 파일 내 오프셋 (보통 0)
   - size_in_cont → 새 IDRK 블록 크기
   - fdata_id     → 새 fdata_id
4. RDB 헤더는 변경 불필요 (엔트리 수 동일)
```

### 장점

- 크기 제한 없음
- 원본 .fdata 파일 불변 (백업 부담 적음)
- 동작 원리가 명확함 (엔진이 RDB를 신뢰한다면 확실히 작동)

### 단점

- RDB 자체를 수정해야 함 (원본 RDB 백업 필요)
- RDB 수정 도구가 필요

### 검증 포인트

엔진이 RDB를 파싱할 때 엔트리의 `entry_size`를 기반으로 순차 탐색하는지,
아니면 빌드된 해시맵을 사용하는지 확인 필요.
수정 시 엔트리 내부 크기는 변경하지 않으므로 이 부분은 안전하다.

---

## 전략 C: RDB 중복 추가 (Last-wins 가설)

RDB의 끝에 동일한 `file_ktid`를 가진 새 엔트리를 추가한다.
엔진이 해시맵에 "나중에 오는 엔트리로 덮어쓰기" 방식을 사용한다면 작동한다.

### 절차

```
1. 새 모드 .fdata 파일 생성
2. RDX에 새 fdata_id 등록
3. RDB 끝에 새 엔트리 추가:
   - file_ktid = 교체할 에셋의 file_ktid (기존과 동일)
   - fdata_id, fdata_offset = 새 파일 정보
4. RDB 헤더의 file_count += 추가된 수
```

### 장점

- 원본 RDB 엔트리를 건드리지 않음
- 상대적으로 단순
- 쉽게 모드를 "비활성화" 가능 (추가한 엔트리만 제거)

### 검증 필요

```
검증 방법:
1. 임의 에셋(예: 특정 텍스처)을 duplicate 등록
2. 게임 실행 후 두 번째 엔트리의 에셋이 사용되는지 확인
3. 또는 엔진 바이너리에서 RDB 파싱 로직 리버싱
```

엔진이 `std::unordered_map::insert` (first-wins) 방식이면 실패.
`operator[]` (last-wins) 방식이면 성공.

---

## 전략 D: 완전 새 에셋 등록

기존 에셋과 완전히 다른 `file_ktid`를 가진 새 에셋을 RDB에 추가한다.
게임 내 콘텐츠를 순수하게 추가(추가 캐릭터, 추가 의상 등)할 때 필요한 방식.

### 절차

```
1. 새 file_ktid 결정 (기존과 충돌하지 않는 값)
2. 새 에셋들 제작 (G1M, G1T, .ktid, kidsobjdb)
3. RDB + RDX에 모든 새 에셋 등록
4. 씬 DB(kidsscndb) 또는 캐릭터 선택 코드가 새 에셋을 참조하도록 수정
```

### 현실적 어려움

- 씬 DB나 캐릭터 선택 코드는 게임 exe나 별도의 DB 파일에 하드코딩되어 있을 가능성 높음
- KatanaEngine이 RDB에서 file_count 외에 에셋 목록을 검증하는지 알 수 없음
- 새 에셋이 참조되는 진입점을 찾기 어려움

### 현실적 용도

- 씬 내 기존 오브젝트를 새 에셋으로 완전 교체하되 FileKtid만 새로 만들고
  kidsobjdb를 수정해서 연결하는 방식
- 혹은 에셋 자체가 아니라 씬 스크립트 레벨에서 참조를 교체하는 방식

## 추천 전략 조합 (범용 모드 매니저 기준)

### Phase 1: 전략 B (RDB 리다이렉트) 구현

가장 신뢰성 높고 크기 제한 없는 방식. 모드 매니저의 핵심.

```
모드 패키지 구조:
  mymod/
    mod.json           ← 교체 매핑 (file_ktid → 내부 파일 경로)
    assets/
      0xABCD1234.g1m   ← 교체할 에셋 원본 (압축 전)
      0x11223344.g1t
      ...
```

매니저 동작:
1. 모든 활성 모드의 `mod.json` 수집
2. 각 에셋을 zlibext 압축 → IDRK 블록 구성 → 단일 `mods.fdata`에 packed
3. `mods.fdata`를 RDX에 새 fdata_id로 등록
4. RDB의 해당 file_ktid 엔트리들의 위치 메타를 `mods.fdata` 오프셋으로 갱신
5. 패치된 RDB를 원본과 분리된 파일로 저장 (원본 불변)

### Phase 2: 전략 C 검증 후 대체

Last-wins가 검증되면 RDB 엔트리 수정 없이 끝에 추가하는 방식으로 단순화.

## 위험 요소

| 위험 | 설명 | 완화책 |
|------|------|--------|
| 해시 검증 | 엔진이 IDRK 블록의 CRC/해시를 검증할 수 있음 | 실제로 검증하는지 테스트 필요 |
| fdata_id 고갈 | u16 범위(0~65535) 초과 불가 | DOA6 기준 실제 사용 범위 확인 |
| RDB 정렬 가정 | 엔진이 file_ktid 기준 정렬을 전제하고 이진탐색할 경우 중복 엔트리 문제 | 엔진 바이너리 확인 |
| KTID 체인 불일치 | G1M 교체 시 .ktid/G1T 체인이 새 G1M과 맞지 않음 | 모드 패키지에 체인 전체 포함 |
